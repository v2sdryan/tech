03. 2026-03-11｜Atlassian 建立了一個工具來將 Jira 用戶遷移到雲端，但它使遷移速度變慢

全文翻譯（繁體中文・香港）

Atlassian 承認，它開發的用於將 Jira 用戶轉移到雲端的工具實際上比完成相同工作的舊程式碼慢，而且它為加快速度所做的努力也存在速度問題。

這家澳洲協作軟體公司去年決定停止其資料中心產品，並將用戶轉移到雲端產品。這項決定是在 Atlassian 終止其伺服器產品五年後做出的。

正如高級軟體工程師 Priyansh Jain 在周二的帖子中所解釋的那樣，Atlassian 經營著一個遷移平台團隊，該團隊在 API 驅動的架構上建立了遷移管道。

「然而，這種架構被證明是阻塞的且可擴展性較差，而且我們遷移管道中的客戶太大，無法使用這種方法進行遷移，」Jain 寫道。

因此，Atlassian 建立了一個新的遷移架構，Jain 表示該架構「以簡化的方式運行，並巧妙地避免了 API 驅動架構中存在的瓶頸和可擴展性問題」。該公司將其發布給需要將 Jira 實施遷移至多 20,000 個席位的客戶。

但當 Atlassian 對其進行測試時，該公司發現新的遷移管道比以前的工具花費了大約 34% 的時間，並且在綜合測試中，總體工作項吞吐量下降了大約 60%。 」

Jain 寫道：“對於擁有數萬名用戶和龐大專案組合的客戶來說，解決這個問題是沒有商量餘地的。”

「我們對不同的工作節點大小和配置進行了基準測試。最初的設置在小型節點上運行；擴大它們的規模可以顯著提高吞吐量，平衡成本與性能，」Jain 的帖子指出。他說 Atlassian 還收緊了自動擴展規則，以便每當 CPU 使用率激增時，工作節點就會快速啟動，從一開始就保持高吞吐量。 」

Atlassian 還「發現了輪詢逾時的錯誤配置。我們的工作項目處理通常需要 60-120 秒，但消費者超時設定為 40 秒。這意味著批次會在運行中重試，浪費工作並將吞吐量削減 30-40%。設定一個實際的超時（300 秒）可以立即解決問題。」

經過多次錯誤修復後，Atlassian 推出了一款工具，可以將大型多專案遷移的中位數吞吐量提高約 6 倍。

Jain 的貼文沒有說明 Atlassian 花了多長時間扭轉局面，但確實表示該公司還開始開發工具來遷移多達 50,000 個席位的更大 Jira 實例，

“目標很明確：在 36 小時內遷移數千個項目（大約 6,500 個）和多達約 750 萬個工作項目，導入階段在約 24 小時內完成，”Jain 透露，並補充道，“我們不能只是‘將旋鈕調到 11’並希望得到最好的結果。”

但我們離題了：Jain 詳細介紹了創建新工具遇到的問題，例如遷移開始緩慢，並且在 45-60 分鐘後才加速，因為 Atlassian 將自動縮放邏輯與 CPU 負載聯繫在一起。該公司透過 Jain 所描述的“一種機制來解決這個問題，該機制可以在重大遷移開始時主動確保最低數量的工作節點正在運行。導入開始全力進行，將總體遷移時間縮短了 30-60 分鐘。”

這項變更也為 Atlassian 帶來了回報，因為它使該公司能夠保持較小的穩態足跡，並僅在需要時進行擴展，從而每月減少基礎設施成本高達 65,000 美元。

另一個問題是大規模遷移導致 API 錯誤，“因為我們資料庫叢集的唯讀副本難以跟上繁重的寫入負載。”

Atlassian 解決了所有問題，這次驗證了該工具是否已準備好為 50,000 名客戶提供服務。
