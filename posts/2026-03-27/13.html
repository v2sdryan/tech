13. 2026-03-26｜使用 AI 編碼並不意味著您的程式碼更安全

全文翻譯（繁體中文・香港）

隨著越來越多的人使用人工智慧工具編寫程式碼，這些工具本身也引入了更多的漏洞。

佐治亞理工學院 SSLab 的研究人員一直在追蹤人工智慧產生的程式碼中的缺陷所引起的 CVE。

去年 8 月，他們只發現了兩個可以明確連結到 Claude Code 的 CVE：CVE-2025-55526（n8n-workflows 中的一個嚴重性為 9.1 的目錄遍歷漏洞）和 GHSA-3j63-5h8p-gf7c（x402 SDK 中的一個不正確的輸入錯誤）。

3 月份，他們確定了 35 個 CVE，其中 27 個由 Claude Code 編寫，4 個由 GitHub Copilot 編寫，2 個由 Devin 編寫，Aether 和 Cursor 各 1 個。

克勞德·科德 (Claude Code) 的代表性過高似乎是由於其最近人氣飆升所致。根據軟體工程師 Jodan Alberts 創建的分析網站 Claude's Code 的數據，在過去 90 天裡，Claude Code 已向公共儲存庫添加了超過 307 億行程式碼。

佐治亞理工學院的研究人員於 2025 年 5 月 1 日開始測量，截至 2026 年 3 月 20 日，CVE 記分卡顯示：

在分析的 43,849 個建議中，有 74 個 CVE 可歸因於 AI 編寫的程式碼。

佐治亞理工學院 SSLab 的研究員趙漢清在一封電子郵件中告訴 The Register，這些 AI CVE 可以被視為一個下限，而不是一個比率。

他說：「在這 74 個案例中，我們發現了明確的證據，表明人工智慧產生的程式碼導致了該漏洞。」 「這並不意味著其他約 50,000 個案例是人類編寫的。這意味著我們無法檢測到人工智慧參與這些案例。

「以 OpenClaw 為例，它有超過 300 條安全建議，而且似乎經過了嚴格的 Vivi 編碼，但大多數 AI 痕跡已被剝離。我們只能自信地確認大約 20 個具有清晰 AI 信號的案例。基於此類項目，我們估計真實數字可能比我們目前檢測到的數字高 5 到 10 倍。」

趙表示，CVE 數量不應僅僅因為數量較少就被視為人工智慧程式碼工具可以提供更安全程式碼的標誌。

「現在，僅 Claude Code 就出現在 GitHub 上超過 4% 的公開提交中，」他解釋道。 「如果人工智慧確實只對 50,000 個公共漏洞中的 74 個負責，那就意味著人工智慧生成的程式碼比人類編寫的程式碼安全幾個數量級。我們認為這是不可信的。”

他說，較低的數字“反映了檢測盲點，而不是卓越的人工智慧程式碼品質。”

喬治亞理工學院的研究結果強化了喬治城大學安全與新興科技中心在 2024 年 11 月發表的研究成果。
